# frontend-module-api-digest

调研任意前端视图中的功能模块，把每个模块展示的字段（label / 字段 key / 列表 column / 筛选项 / 统计指标）和它们对应的查询接口梳理成结构化 Markdown。

## 适用场景

不限视图类型，常见包括：

- **详情页 / 列表页** - label + 字段 key / 列表 column
- **Dashboard / 工作台** - 统计卡片的指标与取值字段
- **图表** - 维度与序列字段
- **弹窗 / 抽屉** - 浮层内的展示模块
- **筛选条件区** - 筛选项 + 控件 + 选项来源

## 核心约定

- **只汇总查询接口**（`GET` 及查询型 `POST`），剔除增删改，除非用户明确要求。
- **给接口地址，不给代码里的方法名** - 输出 `GET /api/xxx/yyy`，而非 `getApiXxxYyy`。
- **精确区分字段真实来源** - 同一模块的字段可能来自多个接口，必须读组件源码确认，存疑标注“待核实”。
- **完整追踪 code -> name 映射链路** - 查清名称来自另一个列表接口、前端硬编码常量、store 还是 hook，并追到最上游。
- **按视图类型选择字段口径** - 列表用 column、筛选用 key+控件+选项、图表用指标+维度、详情用 label+key。

## 工作方式

1. 与用户确认视图范围，避免漂移。
2. 派 Explore/general-purpose 子代理广度搜索、并行调研独立模块，完整读取组件源码抽取字段与数据获取调用。
3. 在 `services/`（或等价目录）还原真实的 HTTP method + path，剔除写操作。
4. 追踪每个字段的来源与 code->name 映射链路。
5. 输出：每个模块先给主查询接口再给字段表，最后给一张全局“查询接口汇总表”。

## 触发示例

- “调研/汇总某某页面的模块和接口”
- “这个字段来自哪个接口”
- “这些 code 怎么展示成名称”
- “筛选条件调了哪些接口”
- “Dashboard 这个卡片的数据从哪来”

即使没有出现“汇总接口”字样，只要是在梳理某个视图/模块的展示内容与数据来源，就应使用本 skill。

## 与 detail-module-api-digest 的关系

本 skill 是后者的泛化版本：去掉了“详情页/列表页”限定，把同样的梳理能力扩展到任意前端视图（Dashboard、弹窗、抽屉、筛选表单、图表卡片等）。原 skill 的核心原则、工作流程与输出风格均保留。
